岗位配置包系统架构 v2.0

从通用角色模型到财税行业精准岗位适配。基于真实组织架构的四层继承系统，让每个岗位一键获得量身定制的 AI Coding Agent 工作环境。

v2.0 -- 公司组织适配版 10 岗位 / 4 职能线 / 15 内容层 2026-03-26

Executive Summary

基于财税 AI-Agent 平台公司的真实组织架构（2 中心、8 部门、20+ 岗位组），本文档定义了一套四层继承的岗位配置包系统。核心架构决策：财税行业知识作为 L0 层注入所有岗位（而非仅合规岗位），因为该公司所有员工——从前端到算法——都在财税领域工作。系统采用 2 表数据模型（pack_layers + config_packs），构建时预生成 50 个静态文件，通过问题树 UX 引导用户在 2 步内找到最匹配的配置包。

Section 02

四层继承模型

每个岗位配置包由 3 层内容块拼接而成（L0 + L1 + L2），可选的 L3 由用户在安装后自行覆盖。层级越高优先级越高——当规则冲突时，角色层覆盖职能线层，职能线层覆盖行业基础层。

L3
项目上下文层 (可选) — 用户安装后自行添加。项目特定的技术栈、代码规范、团队约定。
示例：APP 项目部用 React Native + Go，PC 数字项目部用 Vue + Java
L2
岗位角色层 — 10 个角色各一个。包含该岗位的质量优先级、专属工具链、领域规则、启动提示词。
示例：frontend-engineer 的 React 19 / Tailwind v4 / CWV 目标 / WCAG 2.1 / 组件优先
L1
职能线层 — 4 条线各一个。编码规范、协作流程、共享工具、安全基线。
示例：line-engineering 的 Git 工作流 / PR 格式 / Conventional Commits / Code Review 规范
L0
财税行业基础层 (全员) — 所有 10 个岗位都继承。增值税发票规则、金税系统 API 模式、会计准则意识、合规红线、财税术语表。
核心洞察：财税公司的前端工程师需要懂发票 UI 规则，后端需要懂金税接口，测试需要懂合规校验。行业知识不是某个岗位的专属。
关键架构决策

财税知识放 L0（全员继承），不放 L2（仅合规岗位）。 原因：该公司所有员工都工作在财税领域。前端工程师需要懂发票 UI 模式（品名/税率/金额对齐），后端需要懂金税四期 API 对接模式，大数据工程师需要懂税务数据脱敏规则。行业领域知识是共有语言，不是某个角色的专属知识。

合并策略

文件类型 合并方式 说明
CLAUDE.md 字符串拼接 L0 + L1 + L2 按顺序拼接，分隔符 \n\n---\n\n。不做 Markdown AST 合并。简单、可预测、可审计
AGENTS.md 字符串拼接 同上。Agent 定义按层级叠加
settings.json 深度合并 MCP 服务器列表做 union 合并。L2 角色层的设置覆盖 L1/L0 的同名 key
prompts.md 仅 L2 提示词是角色专属的，不继承。每个角色 4-6 个场景化 prompt
install.sh 模板生成 统一模板，填入 pack-id。下载已合并的最终文件，不做运行时合并
Section 03

数据架构：2 表模型

采用 Rich Hickey 的值不可变思想：What（内容）与 How（组合）分离。pack_layers 存储不可变的内容块，config_packs 存储组合引用。生成脚本在构建时读取两表、合并输出静态文件。

表 1：pack_layers（内容块）

CREATE TABLE pack_layers ( id TEXT PRIMARY KEY, -- 'universal' | 'line-engineering' | 'role-frontend' ... type TEXT NOT NULL, -- 'universal' | 'line' | 'role' name TEXT NOT NULL, -- 'Finance-Tax Industry Base' name_zh TEXT NOT NULL, -- '财税行业基础' content_claude_md TEXT DEFAULT '', content_agents_md TEXT DEFAULT '', content_settings TEXT DEFAULT '{}', -- JSON: MCP servers config content_prompts_md TEXT DEFAULT '', sort_order INTEGER DEFAULT 0, created_at TEXT DEFAULT (datetime('now')) ); -- 15 rows: 1 universal + 4 line + 10 role

表 2：config_packs（组合引用）

CREATE TABLE config_packs ( id TEXT PRIMARY KEY, -- 'frontend-engineer' name TEXT NOT NULL, -- 'Frontend Engineer' name_zh TEXT NOT NULL, -- '前端工程师' description TEXT DEFAULT '', description_zh TEXT DEFAULT '', icon TEXT DEFAULT 'person', color TEXT DEFAULT '#003ea8', line TEXT NOT NULL, -- 'engineering' | 'data-ai' | 'product' | 'business' line_zh TEXT NOT NULL, -- '研发职能线' layer_ids TEXT NOT NULL, -- '["universal","line-engineering","role-frontend"]' version TEXT DEFAULT '1.0', download_count INTEGER DEFAULT 0, created_at TEXT DEFAULT (datetime('now')), updated_at TEXT DEFAULT (datetime('now')) ); -- 10 rows, each referencing 3 layer IDs

Why 2 Tables, Not 1

单表模型（v4 方案）将所有内容塞入 config_packs 的 claude_md/agents_md 字段。问题：无法复用。5 个研发岗位的 Git 工作流规范完全相同，但单表模型下必须复制 5 份。2 表模型下，line-engineering 层只写一次，5 个岗位引用同一个 layer ID。修改一处，5 个包同时更新。

Section 04

生成管线

scripts/generate-packs.ts 是核心纯函数。读取种子 SQL（或 D1 查询），按 layer_ids 顺序合并每层内容，输出到 web/public/packs/。

伪代码

function generatePack(pack: ConfigPack, layers: Map<string, PackLayer>): PackFiles { const orderedLayers = pack.layer_ids.map(id => layers.get(id)!); return { // CLAUDE.md: 字符串拼接 (L0 + L1 + L2) claudeMd: orderedLayers .map(l => l.content_claude_md) .filter(Boolean) .join('\n\n---\n\n'), // AGENTS.md: 字符串拼接 agentsMd: orderedLayers .map(l => l.content_agents_md) .filter(Boolean) .join('\n\n---\n\n'), // settings.json: 深度合并 (后层覆盖前层) settings: orderedLayers.reduce( (merged, l) => deepMerge(merged, JSON.parse(l.content_settings)), {} ), // prompts.md: 仅最后一层 (L2 角色层) prompts: orderedLayers.at(-1)!.content_prompts_md, // install.sh: 模板填充 install: generateInstallScript(pack.id), }; }

构建时生成，非运行时解析

Layer 合并在 npm run build 阶段完成，输出为纯静态文件。用户下载的是已合并的最终产物。这意味着：(1) CDN 可以直接缓存；(2) install.sh 不需要合并逻辑；(3) 离线可用；(4) 构建失败 = 部署失败，不会交付半成品。

Section 06

关键架构决策

以下决策来自 5 路蜂群头脑风暴（Drucker / Jobs / Godin / Munger / Hickey）和 3 路架构评审（PM / Hickey / Jobs）。每条决策记录了选择、理由和否决项。

ADR-01
财税知识放 L0，不放 L2
所有岗位继承行业基础层。财税公司的前端也需要懂发票 UI，后端也需要懂金税 API。
否决项：仅合规岗位注入（会导致其他岗位交付的代码不懂行业规则）
ADR-02
2 表模型，不用单表
pack_layers + config_packs 分离内容与组合。修改一个层自动影响所有引用它的包。
否决项：单表（v4 方案，内容无法复用，5 个研发岗需复制 5 份相同规范）
ADR-03
字符串拼接，不做 AST 合并
CLAUDE.md 按层级直接拼接，中间用 --- 分隔。简单、可预测、可审计。
否决项：Markdown AST 合并（复杂度高，标题层级冲突难解）
ADR-04
构建时预生成，不做运行时解析
generate-packs.ts 在 build 阶段输出静态文件。CDN 缓存、离线可用、构建即验证。
否决项：运行时合并（CDN 不可缓存，install.sh 需合并逻辑，增加复杂度）
ADR-05
问题树 UX，不用卡片网格
2 步引导 + 1 个推荐。降低认知负荷，提升转化率。保留"查看全部"作为退路。
否决项：20 张卡片网格（选择焦虑，新用户根本不知道自己要什么）
ADR-06
按职能分组，不按组织架构
APP 前端、PC 前端、财税前端共用同一个 pack。技能栈相同则配置相同。
否决项：按部门 1:1 映射（会导致内容高度重复，维护成本激增）
ADR-07
先 10 个包，不做 22 个
覆盖该公司全部核心岗位。通用版 22 角色中的 Creative/Management 线暂不做。
否决项：一次铺满 22 角色（工作量大，部分角色 Skills/MCP 缺口大）
ADR-08
7 项不做清单
不做版本管理、不做在线编辑器、不做 RBAC、不做 Pack 市场、不做自定义合并、不做级别变体、不做 Pack 对比。
原则：最小必要复杂度。每一项都可以在 v3+ 按需添加

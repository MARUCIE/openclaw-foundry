02

数据模型设计

遵循 Hickey 简洁原则：只加 2 张新表 + 1 组 ALTER，拒绝 4 表中间层的过度设计。事件表 append-only，排序字段 cron 物化。

设计原则 (5 路蜂群共识)

01

统一事件流

  • 一张 skill_events 替代 N 张表
  • deploy/review/view/install 全是同一事实结构
  • Append-only，UNIQUE 去重
02

Cron 物化

  • 写入只管写入，不即时更新 skills
  • Cron handler 统一重算衍生字段
  • 解耦写入路径与计算路径
03

内容在 R2

  • README/截图/changelog 存 R2
  • D1 只存元数据指针
  • 不把内容 complect 进关系表

migration-v3.sql

-- [新表 1] 统一事件流
CREATE TABLE IF NOT EXISTS skill_events (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  skill_id TEXT NOT NULL,
  event_type TEXT NOT NULL CHECK (event_type IN (
    'deploy_ok','deploy_fail',
    'review_up','review_down',
    'install','view'
  )),
  fingerprint TEXT NOT NULL,
  tenant_id TEXT DEFAULT '',
  payload TEXT DEFAULT '{}',
  weight REAL DEFAULT 1.0,
  created_at TEXT DEFAULT (datetime('now')),
  UNIQUE(skill_id, event_type, fingerprint)
);
CREATE INDEX IF NOT EXISTS idx_events_skill ON skill_events(skill_id);
CREATE INDEX IF NOT EXISTS idx_events_type ON skill_events(event_type);
CREATE INDEX IF NOT EXISTS idx_events_created ON skill_events(created_at);

-- [新表 2] 策展集合 (平表，JSON skill_ids)
CREATE TABLE IF NOT EXISTS collections (
  id TEXT PRIMARY KEY,
  name TEXT NOT NULL,
  tagline TEXT DEFAULT '',
  description TEXT DEFAULT '',
  skill_ids TEXT NOT NULL DEFAULT '[]',
  curator TEXT NOT NULL,
  cover_type TEXT DEFAULT 'auto',
  featured INTEGER DEFAULT 0,
  install_count INTEGER DEFAULT 0,
  created_at TEXT DEFAULT (datetime('now'))
);
CREATE INDEX IF NOT EXISTS idx_collections_featured
  ON collections(featured DESC, install_count DESC);

-- [skills 表扩展] 衍生字段 (cron 重算)
ALTER TABLE skills ADD COLUMN editorial_tagline TEXT DEFAULT '';
ALTER TABLE skills ADD COLUMN trending_score REAL DEFAULT 0.0;
ALTER TABLE skills ADD COLUMN composite_score REAL DEFAULT 0.0;
ALTER TABLE skills ADD COLUMN review_up INTEGER DEFAULT 0;
ALTER TABLE skills ADD COLUMN review_down INTEGER DEFAULT 0;
ALTER TABLE skills ADD COLUMN stale_penalty REAL DEFAULT 0.0;
为什么不建 skill_combos 中间表？ 249 个 skills 规模下，co-occurrence 推荐只需一个 SQL JOIN (skill_events 自关联)。当 skills 超过 2000 时再考虑物化到 combos 表。过早优化是万恶之源。
数据流

用户行为 → skill_events (append-only) → cron 物化 → skills.composite_score / trending_score → 前端排序。写入路径和计算路径完全解耦。

03

P0-1: composite_score 排序引擎

一行 SQL 变更，让排名从"第三方爬虫分"变成"真实部署质量分"。这是系统动力学 #6 杠杆点（信息流），修复代价最低、影响最大。

公式

composite_score =
    score * 0.35                           -- 第三方信号 (ClawHub 原始分)
  + deploy_success_rate * 100 * 0.30       -- 部署质量 (本站真实数据)
  + (review_up - review_down) * 5 * 0.20   -- 社区评价 (净推荐)
  + trending_score * 0.10                  -- 短期热度 (7 天窗口)
  + (1 - stale_penalty) * 10 * 0.05        -- 活跃惩罚 (非布尔衰减)

trending_score 计算 (时间衰减)

trending_score = SUM(
  CASE event_type
    WHEN 'install' THEN 3.0
    WHEN 'deploy_ok' THEN 2.0
    WHEN 'review_up' THEN 2.0
    WHEN 'view' THEN 0.1
    ELSE 0
  END
  * (1.0 / (1 + julianday('now') - julianday(created_at)))
)
WHERE created_at > datetime('now', '-7 days')

灵感来源：Product Hunt 的投票速率 > 总票数原则。7 天窗口 + 时间衰减确保新 skill 有机会浮出水面，老 skill 不会仅靠历史累积霸榜。

实现路径

1) migration-v3.sql 添加字段 → 2) cron handler 增加重算逻辑 → 3) skills route 排序改用 composite_score DESC。前端无需变更。

04

P0-2: B1 防刷平衡环

Goodhart 定律警告：当 composite_score 成为正式排名依据，它本身就会成为被优化的对象。B1 防刷必须与 R1 质量环同时上线。

防刷机制三层

L1: 指纹去重

UNIQUE(skill_id, event_type, fingerprint) 已内置。同一用户对同一 skill 同一行为只计一次。

L2: 权重衰减

同一 ASN 段 24h 内 > 5 个不同 skill 的同类事件，后续事件 weight 降至 0.1。用 fingerprint 前 8 位做 ASN 段近似。

L3: 评价身份门槛

review_up/review_down 事件强制要求 tenant_id 非空。行为信号 (deploy/view/install) 允许匿名。

风险：过度防刷会伤害正常用户体验。L2 权重衰减的阈值 (5 事件/24h) 需要通过真实数据校准。初期设置为仅记录 (weight=1.0 但打标记)，积累 2 周数据后再启动实际降权。
05

P1-1: Skill 详情页 (7 区布局)

从工具文档变成产品页。开发者打开 skill 页面时真正问的是"用了它，我能做什么以前做不到的事" -- 当前 UI 完全没有回答这个问题。
区域 内容 数据来源 优先级
1. Hero 名称 + editorial_tagline (80 字) + 最佳 GIF/截图 skills.editorial_tagline + R2 P1
2. 场景 "用了能做什么" -- 3 个 bullet，不是功能而是场景 payload JSON 或 R2 readme P1
3. Combo "经常搭配" 横向卡片 (2-3 个) co-occurrence SQL query P1
4. 证词 战绩证词 (问题 + 之前耗时 + 之后耗时)，最多 3 条 skill_events review_up + payload P1
5. 安装 一键安装 [STICKY] -- 7 平台选择 + 命令复制 现有 INSTALL_TARGETS DONE
6. 权限 permission_manifest 人话版 skills.permission_manifest P1
7. 活跃度 30 天 sparkline + 最后更新时间 + stale 状态 skill_events 聚合 + skills.stale P2
关键设计原则 (Jobs)：安装命令永远是 sticky 的。用户在看完所有内容后只需要一个动作，不要让他们滚动回顶部找按钮。
editorial_tagline 策略

Top 50 个 S/A 级 skill 由管理员手动撰写 tagline。其余自动截取 description 前 80 字。一个字段的差异，让详情页从工具文档变成产品页。

Collection 的五种类型 (Godin)

工作流叙事型

"我是如何用这 7 个 skill 在两周内独立完成一个 SaaS 的" -- 叙事才能传播。

替代方案型

"用这 5 个 skill 替代 Copilot 的 80% 功能，成本是零" -- 这种标题会被截图分享。

防坑指南型

"我踩过的 6 个坑，以及修复它们的 skill" -- 负面叙事传播速度是正面的三倍。

角色匹配型

"全栈 solo founder 必备包" / "Tech Lead code review 套装" -- 身份认同驱动转化。

P1-3: 战绩证词系统

开发者不信星级评分。一条 "我用 ontology skill 配合 kuzu-mcp，三小时完成了原本一周的本体建模" 的证词，价值等于 500 个匿名五星。

T1

  • 要求 tenant_id (轻量身份)
  • 结构化问题/时间表单
  • 一条真实证词 = 500 个匿名星

08

反馈回路架构

三个增强环驱动增长，三个平衡环防止系统崩溃。增强环与平衡环必须配对上线。

增强环 (Reinforcing Loops)

R1 质量曝光环 (P0 修复)

deploy_ok 事件 → composite_score 上升 → 排名提升 → 曝光增加 → 更多安装 → 更多事件 → 回到起点

当前状态：断路。deploy_success_rate 未参与 score 计算。修复 = 一行 SQL。

R2 Combo 增强环 (P1 新建)

collection 被安装 → 批量 co-install 事件 → combo 推荐权重上升 → 更多人通过 combo 发现 skill → 更多安装 → 回到起点

启动条件：预置 5+ 个 featured collection 做冷启动。

R3 作者激励环 (P2 新建)

skill 获高评分 → 自动通知 repo owner (GitHub Issue) → 作者有动力维护 → 质量提升 → 评分继续上升

纯信息流激励，不需要账号系统。

平衡环 (Balancing Loops)

B1 防刷环 (P0 同步上线)

大量同源事件 → ASN 异常检测 → weight 降至 0.1 → 刷分收益趋零

B2 陈旧惩罚环 (P0 升级)

stale 从布尔值升级为连续衰减 (stale_penalty REAL)。7 天 fail 率 > 60% 实时触发，不等 cron。

B3 评分通胀抑制 (P2)

S 级 skill 比例超 10% → category 内百分位重算 → 晋级门槛自动收紧 → 比例回落均衡

竞品对标与差异化

7 个成熟市场的对标研究表明：Smithery.ai 发现机制最弱，VS Code Extension Pack 是 Combo 概念的直接验证，Steam 策展人的零激励模式是我们需要避免的反面教材。
维度 Smithery ClawHub VS Code OpenClaw v3
排序算法 不透明 5 维评分 装机数/评分 composite_score (4 维 + 时间衰减)
用户评价 无 基础评分 星级 + 文字 战绩证词 (结构化 before/after)
组合推荐 无 无 Extension Pack Combo 配方 (叙事 + 工作流 + 一键装)
编辑策展 Featured 列表 无 编辑推荐 署名策展人 + 5 种 Collection 类型
趋势热榜 无 无 7 日增长 时间衰减 trending (Product Hunt 模型)
防刷机制 不明 不明 账号验证 三层 (去重 + 权重衰减 + 身份门槛)
竞争壁垒

Combo 配方 + 战绩证词 + 编辑策展署名制 -- 这三个特性的组合在当前所有 AI skill 市场中独一无二。核心壁垒不是技术，是内容积累：一旦有 50 条真实证词和 20 个策展配方，竞品即使抄袭架构也无法复制内容资产。

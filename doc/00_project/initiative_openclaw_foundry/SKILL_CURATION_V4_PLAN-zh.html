Section 03

评分公式设计

当前评级失效的根本原因是没有标准化——直接用原始数值排序，导致高下载量的 Skill 全部挤入 S 级。新方案用对数归一化 + 百分位阈值，保证任何数据分布下都产出正态曲线。

输入归一化 (0-1 区间)

norm_downloads = log10(downloads + 1) / log10(max_downloads + 1) norm_stars = log10(stars + 1) / log10(max_stars + 1) freshness = max(0, 1 - days_since_update / 365) desc_quality = min(1, len(description) / 200) platform_count = len(platforms) / max_platforms

Composite Score

composite = (norm_downloads * 0.3) + (norm_stars * 0.25) + (freshness * 0.2) + (desc_quality * 0.15) + (platform_count * 0.1)

评级阈值 (百分位驱动)

评级 百分位 约数量 含义
STop 5%~100行业标杆，值得每个人安装
A5-20%~300优秀，特定场景强烈推荐
B20-60%~800可用，满足基本需求
C60-90%~600有价值但有局限性
DBottom 10%~200低质量或已过时
Why 百分位而非固定阈值

固定阈值（如 score>0.8 为 S）会随数据变化而膨胀或收缩。百分位保证无论数据质量如何变化，S 级永远是 top 5%。这是 Netflix/Spotify 等平台的标准做法。

Section 05

关键决策

ADR-01
百分位评级，不用固定阈值
保证正态分布不受原始数据质量波动影响。S 级永远是 top 5%。
否决：固定阈值（会随数据变化膨胀/收缩，当前就是这样坏的）
ADR-02
LLM 批量打标，不手动标注
2000 个 Skill 手动标注不现实。Gemini Flash 成本 $0.02，S/A 级人工复审。
否决：纯手动（无法扩展）/ 纯规则（准确率低）
ADR-03
{author}/{name} 去重，不用 hash
匹配 npm/GitHub 命名惯例，保留来源信息，可读性强。
否决：hash 去重（失去可读性）/ 纯名称去重（不同作者可能同名）
ADR-04
3 维标签，不用扁平列表
tech-stack / scenario / platform 覆盖发现的三个轴——用什么技术、做什么事、在哪里用。
否决：单一扁平标签（维度混淆，搜索效果差）
ADR-05
先 P0 再 P1 再 P2，不一次全做
去重和评级是基础——不修好这两个，标签和编辑内容建在废墟上。
否决：并行全做（依赖关系决定顺序，评级要用去重后的数据）
ADR-06
每日质量报告，不默默同步
Skill 质量会持续退化。每日报告让问题可见，建立信任。
否决：静默同步（问题积累，发现时已成灾）
